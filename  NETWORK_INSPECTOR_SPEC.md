# Network Inspector Spec

## Overview

This module provides a rich in-app network inspector for iOS applications running in lower environments such as:
- development
- QA
- staging
- internal beta

The inspector captures network activity in real time and displays it inside the app for debugging and validation.

The initial implementation will live directly inside the app codebase, but it must be designed so it can later be extracted into a library or Swift Package with minimal changes.

---

## Goals

### Main goals
- Capture HTTP/HTTPS traffic generated by app networking
- Show request and response details in real time
- Display transport and timing metrics
- Provide a polished debugging UI
- Keep the design modular and extractable
- Make it safe for non-production internal use

### Non-goals
- Full packet sniffing
- Production analytics
- Capturing all OS-level traffic outside app networking stack
- Replacing Instruments / proxy tools

---

## Core Functional Requirements

## 1. Capture Model

Each captured entry should include at minimum:

### Identity
- unique id
- sequence number if useful

### Request
- URL
- host
- path
- query items
- HTTP method
- headers
- body data
- body text preview
- content type
- content length

### Response
- status code
- headers
- body data
- body text preview
- content type
- content length

### Error
- localized description
- domain/code where applicable

### Timing
- request start timestamp
- request end timestamp
- total duration in milliseconds

### Metrics
When available from `URLSessionTaskMetrics`, capture:
- task interval
- transaction metrics list
- redirect count
- resource fetch type
- reused connection
- proxy use if available
- network protocol name if available
- dns timing
- connect timing
- secure connection timing
- request timing
- response timing

### State
- in progress
- completed
- failed
- cancelled if distinguishable

---

## 2. Interception Strategy

Use a custom `URLProtocol` implementation.

### Requirements
- Intercept requests created by monitored `URLSession`
- Forward requests safely
- Capture response data incrementally or at completion
- Avoid infinite loops by tagging handled requests
- Preserve original request behavior as much as possible

### Notes
The interception layer must be careful with:
- duplicate handling
- redirects
- streaming
- body capture limits
- thread safety

---

## 3. Metrics Strategy

Metrics should be captured through session delegate hooks using `URLSessionTaskMetrics`.

### Requirements
- Associate metrics with captured request records
- Safely update records when metrics arrive after completion
- Expose metrics in a UI-friendly format
- Provide derived durations for display

---

## 4. Storage

Implement a thread-safe store.

### Store requirements
- observable for UI updates
- supports insert/update/remove/clear
- supports max retention count
- removes oldest records when over limit
- supports filtering/search on UI side or query helper side
- optional persistence to disk

### Persistence
Optional lightweight persistence can store a recent session snapshot as JSON.

Persistence should be:
- best-effort
- non-blocking
- easy to disable
- bounded in size

---

## 5. Redaction

Sensitive values must be redacted by default.

### Sensitive header keys
- Authorization
- Cookie
- Set-Cookie
- x-api-key
- api-key

### Sensitive body keys
- token
- access_token
- refresh_token
- password
- secret
- client_secret

### Redaction behavior
- case-insensitive matching
- replace sensitive values with a visible placeholder like `██ REDACTED ██`
- support extension via config
- apply before persistence/export/UI display

---

## 6. Body Capture Rules

The inspector must support body capture limits.

### Request/response body behavior
- if body is below configured max size, store body
- if body exceeds size limit, store truncated preview and metadata
- if binary, avoid rendering as text unless safe
- if JSON, provide pretty-printed view
- if text, provide raw text view
- if multipart/form-data, show metadata and optionally partial text preview
- if image/file/blob, show metadata only unless explicitly safe to preview

### Detection
Use:
- content-type
- MIME type heuristics
- text decoding heuristics

## 8. Configuration API

Expose a public configuration model.

### Suggested config fields
- `isEnabled`
- `environmentName`
- `allowedHosts`
- `ignoredHosts`
- `maxStoredRequests`
- `maxCapturedBodyBytes`
- `captureRequestBodies`
- `captureResponseBodies`
- `captureMetrics`
- `persistToDisk`
- `showsFloatingOverlay`
- `allowsExport`
- `redactedHeaderKeys`
- `redactedBodyKeys`

### Behavior
- if disabled, inspector should no-op cleanly
- if allowed hosts is non-empty, only capture matching hosts
- ignored hosts should always win over allowed hosts

## 9. Public API

Design a lightweight entry point, for example:

- configure

- stop
- clear
- export
- attach overlay

## 10. Export

Support:
- export single request as cURL
- export all captured requests as JSON

### cURL export should include
- URL
- method
- headers
- request body when safe
- proper escaping as reasonable

### JSON export should include
- sanitized entries
- timestamps
- status/error details
- metrics when available

## 11. Error Handling

The module should fail safely.

### Requirements
- no crashes due to malformed payloads
- graceful fallback for unsupported body types
- safe handling for nil response/body
- safe handling for invalid encoding
- no hard dependency on persistence success

## 12. Library Extraction Readiness

Design choices should make future extraction simple.

### Expectations
- avoid app-specific naming
- keep public API cohesive
- isolate UI from capture engine where possible
- reduce cross-layer coupling
- keep configuration centralized
- avoid assumptions about app root structure

## 15. Acceptance Criteria

The implementation is acceptable if it:
- compiles as normal Swift source files
- captures requests and responses in a monitored session
- shows logs in a real-time in-app UI
- displays headers, bodies, and timing info
- redacts secrets by default
- supports searching/filtering
- is organized so it can later be moved into a library
## Additional interception requirement

This inspector must capture traffic not only from direct `URLSession` usage, but also from networking wrappers and abstractions such as Alamofire and similar libraries, provided they ultimately use `URLSession`.

### Requirements
- The interception layer must be based on `URLProtocol` and monitored `URLSessionConfiguration`.
- Provide integration helpers for:
  - `.default`
  - `.ephemeral`
  - custom session configurations
- Ensure the design works with Alamofire by allowing developers to inject a monitored `URLSessionConfiguration` into Alamofire `Session`.
- Do not hardcode support for only one client library.
- Architecture should support “any library built on URLSession”.
- Add documentation explaining what is and is not capturable.

### Explicitly document coverage
Covered:
- raw URLSession
- Alamofire sessions using monitored configuration
- custom API clients built on URLSession
- most wrapper libraries built on URLSession

Potentially not covered without extra work:
- WKWebView traffic
- Network.framework sockets
- gRPC/custom transports not using URLSession
- third-party SDKs that create isolated sessions you do not control
- background sessions if not specifically wired for inspection
